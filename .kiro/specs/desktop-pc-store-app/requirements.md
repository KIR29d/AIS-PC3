# Требования к десктопному приложению АИС магазина кастомных ПК

## Введение

Данный документ описывает требования к десктопному приложению на C# для автоматизированной информационной системы управления магазином кастомных ПК. Приложение предназначено для сотрудников магазина и должно обеспечивать управление заказами, сборочными заданиями, складским учетом и аудитом операций.

## Требования

### Требование 1

**Пользовательская история:** Как сотрудник магазина, я хочу войти в систему под своей учетной записью, чтобы получить доступ к функциям в соответствии с моей ролью.

#### Критерии приемки

1. КОГДА пользователь запускает приложение, ТО система ДОЛЖНА отобразить форму авторизации
2. КОГДА пользователь вводит корректные логин и пароль, ТО система ДОЛЖНА предоставить доступ к интерфейсу согласно роли пользователя
3. ЕСЛИ пользователь вводит неверные данные, ТО система ДОЛЖНА отобразить сообщение об ошибке
4. КОГДА пользователь успешно авторизуется, ТО система ДОЛЖНА записать время последнего входа в базу данных

### Требование 2

**Пользовательская история:** Как менеджер, я хочу управлять заказами клиентов, чтобы отслеживать их статус и обеспечивать своевременное выполнение.

#### Критерии приемки

1. КОГДА менеджер открывает модуль заказов, ТО система ДОЛЖНА отобразить список всех заказов с возможностью фильтрации по статусу
2. КОГДА менеджер создает новый заказ, ТО система ДОЛЖНА позволить выбрать клиента и добавить компоненты
3. КОГДА менеджер изменяет статус заказа, ТО система ДОЛЖНА обновить информацию в базе данных и записать изменение в аудит
4. ЕСЛИ заказ переводится в статус "отгружен", ТО система ДОЛЖНА автоматически установить дату отгрузки

### Требование 3

**Пользовательская история:** Как сборщик, я хочу видеть сборочные задания и управлять процессом сборки, чтобы эффективно выполнять свою работу.

#### Критерии приемки

1. КОГДА сборщик открывает модуль сборки, ТО система ДОЛЖНА отобразить список заданий со статусом "ожидает" и "в_сборке"
2. КОГДА сборщик берет задание в работу, ТО система ДОЛЖНА изменить статус на "в_сборке" и записать время начала
3. КОГДА сборщик завершает сборку, ТО система ДОЛЖНА изменить статус на "готово" и записать время завершения
4. КОГДА сборщик просматривает задание, ТО система ДОЛЖНА отобразить полный список необходимых компонентов с количеством

### Требование 4

**Пользовательская история:** Как сотрудник склада, я хочу управлять остатками компонентов, чтобы контролировать наличие товаров и избегать дефицита.

#### Критерии приемки

1. КОГДА сотрудник склада открывает модуль склада, ТО система ДОЛЖНА отобразить текущие остатки всех компонентов
2. КОГДА сотрудник обновляет количество компонента, ТО система ДОЛЖНА проверить корректность данных и обновить остатки
3. ЕСЛИ количество компонента становится меньше минимального порога, ТО система ДОЛЖНА выделить его в списке
4. КОГДА происходит изменение остатков, ТО система ДОЛЖНА записать операцию в журнал аудита

### Требование 5

**Пользовательская история:** Как администратор, я хочу управлять пользователями и их ролями, чтобы контролировать доступ к системе.

#### Критерии приемки

1. КОГДА администратор открывает модуль пользователей, ТО система ДОЛЖНА отобразить список всех пользователей с их ролями
2. КОГДА администратор создает нового пользователя, ТО система ДОЛЖНА потребовать заполнения всех обязательных полей
3. КОГДА администратор изменяет роль пользователя, ТО система ДОЛЖНА обновить права доступа немедленно
4. КОГДА администратор деактивирует пользователя, ТО система ДОЛЖНА запретить ему вход в систему

### Требование 6

**Пользовательская история:** Как администратор, я хочу просматривать журнал аудита, чтобы отслеживать все изменения в системе.

#### Критерии приемки

1. КОГДА администратор открывает модуль аудита, ТО система ДОЛЖНА отобразить все записи журнала с возможностью фильтрации
2. КОГДА происходит любое изменение данных, ТО система ДОЛЖНА автоматически создать запись в журнале аудита
3. КОГДА администратор просматривает запись аудита, ТО система ДОЛЖНА показать детали изменения в читаемом формате
4. ЕСЛИ администратор фильтрует записи по дате, ТО система ДОЛЖНА отобразить только записи за указанный период

### Требование 7

**Пользовательская история:** Как пользователь системы, я хочу работать с интуитивно понятным интерфейсом в серых тонах, чтобы комфортно выполнять свои задачи.

#### Критерии приемки

1. КОГДА пользователь работает с приложением, ТО интерфейс ДОЛЖЕН использовать серую цветовую схему
2. КОГДА в интерфейсе требуются изображения, ТО система ДОЛЖНА отображать заглушки с подписями
3. КОГДА пользователь взаимодействует с элементами управления, ТО они ДОЛЖНЫ обеспечивать визуальную обратную связь
4. ЕСЛИ пользователь работает с формами, ТО они ДОЛЖНЫ быть логично организованы и легко читаемы
