# Требования к десктопному приложению магазина кастомных ПК

## Введение

Данный документ описывает требования к автоматизированной информационной системе управления продажами и сборкой кастомных ПК (АИС «Кастом ПК»). Система представляет собой десктопное приложение на C# (.NET 8) с интеграцией к базе данных MySQL, предназначенное для автоматизации процессов оформления заказов, управления складскими остатками и контроля выполнения заказов.

## Требования

### Требование 1: Система аутентификации и авторизации

**Пользовательская история:** Как пользователь системы, я хочу войти в систему под своей учетной записью, чтобы получить доступ к функциям согласно моей роли.

#### Критерии приемки

1. КОГДА пользователь запускает приложение, ТО система ДОЛЖНА отобразить форму входа с полями логин и пароль
2. КОГДА пользователь вводит корректные учетные данные, ТО система ДОЛЖНА аутентифицировать пользователя и предоставить доступ к интерфейсу согласно его роли
3. КОГДА пользователь вводит некорректные учетные данные, ТО система ДОЛЖНА отобразить сообщение об ошибке
4. ЕСЛИ пользователь имеет роль "Администратор", ТО система ДОЛЖНА предоставить доступ ко всем модулям
5. ЕСЛИ пользователь имеет роль "Менеджер", ТО система ДОЛЖНА предоставить доступ к управлению заказами и клиентами
6. ЕСЛИ пользователь имеет роль "Сборщик", ТО система ДОЛЖНА предоставить доступ только к сборочным заданиям
7. ЕСЛИ пользователь имеет роль "Клиент", ТО система ДОЛЖНА предоставить доступ к каталогу и оформлению заказов

### Требование 2: Управление каталогом компонентов

**Пользовательская история:** Как администратор, я хочу управлять каталогом компонентов ПК, чтобы поддерживать актуальную информацию о доступных товарах.

#### Критерии приемки

1. КОГДА администратор открывает модуль управления компонентами, ТО система ДОЛЖНА отобразить список всех компонентов с возможностью фильтрации по типу
2. КОГДА администратор добавляет новый компонент, ТО система ДОЛЖНА сохранить информацию о наименовании, типе и производителе
3. КОГДА администратор редактирует компонент, ТО система ДОЛЖНА обновить информацию в базе данных
4. КОГДА администратор удаляет компонент, ТО система ДОЛЖНА проверить отсутствие связанных записей и выполнить удаление
5. ЕСЛИ компонент используется в активных заказах, ТО система ДОЛЖНА запретить его удаление

### Требование 3: Управление складскими остатками

**Пользовательская история:** Как менеджер склада, я хочу отслеживать остатки компонентов на складах, чтобы контролировать наличие товаров.

#### Критерии приемки

1. КОГДА пользователь открывает модуль остатков, ТО система ДОЛЖНА отобразить текущие остатки всех компонентов по складам
2. КОГДА пользователь изменяет количество остатков, ТО система ДОЛЖНА обновить данные с указанием даты изменения
3. КОГДА остаток компонента становится равным нулю, ТО система ДОЛЖНА выделить его цветом для привлечения внимания
4. ЕСЛИ количество остатка отрицательное, ТО система ДОЛЖНА запретить сохранение

### Требование 4: Управление клиентами

**Пользовательская история:** Как менеджер, я хочу управлять информацией о клиентах, чтобы вести учет физических и юридических лиц.

#### Критерии приемки

1. КОГДА менеджер открывает модуль клиентов, ТО система ДОЛЖНА отобразить список всех клиентов с возможностью поиска
2. КОГДА менеджер добавляет нового клиента, ТО система ДОЛЖНА позволить выбрать тип клиента (физическое/юридическое лицо)
3. ЕСЛИ клиент физическое лицо, ТО система ДОЛЖНА запросить ФИО, email и номер телефона
4. ЕСЛИ клиент юридическое лицо, ТО система ДОЛЖНА запросить наименование организации, email и номер телефона
5. КОГДА менеджер сохраняет клиента, ТО система ДОЛЖНА проверить уникальность email адреса

### Требование 5: Оформление и управление заказами

**Пользовательская история:** Как менеджер, я хочу создавать и управлять заказами клиентов, чтобы обеспечить процесс продаж.

#### Критерии приемки

1. КОГДА менеджер создает новый заказ, ТО система ДОЛЖНА позволить выбрать клиента из списка
2. КОГДА менеджер добавляет компоненты в заказ, ТО система ДОЛЖНА автоматически рассчитывать общую сумму
3. КОГДА заказ создается, ТО система ДОЛЖНА установить статус "новый"
4. КОГДА менеджер изменяет статус заказа, ТО система ДОЛЖНА обновить информацию с указанием даты изменения
5. ЕСЛИ заказ переводится в статус "отгружен", ТО система ДОЛЖНА зафиксировать дату отгрузки
6. КОГДА заказ завершается, ТО система ДОЛЖНА обновить остатки компонентов на складе

### Требование 6: Управление сборочными заданиями

**Пользовательская история:** Как сборщик, я хочу видеть назначенные мне задания по сборке ПК, чтобы выполнять свою работу.

#### Критерии приемки

1. КОГДА сборщик входит в систему, ТО система ДОЛЖНА отобразить список его сборочных заданий
2. КОГДА сборщик начинает работу над заданием, ТО система ДОЛЖНА позволить изменить статус на "в_сборке" и зафиксировать время начала
3. КОГДА сборщик завершает сборку, ТО система ДОЛЖНА позволить изменить статус на "готово" и зафиксировать время завершения
4. КОГДА создается сборочное задание, ТО система ДОЛЖНА отобразить список необходимых компонентов с количеством
5. ЕСЛИ у сборщика нет активных заданий, ТО система ДОЛЖНА отобразить соответствующее сообщение

### Требование 7: Административные функции

**Пользовательская история:** Как администратор, я хочу управлять пользователями и системными настройками, чтобы обеспечить корректную работу системы.

#### Критерии приемки

1. КОГДА администратор открывает модуль пользователей, ТО система ДОЛЖНА отобразить список всех пользователей с их ролями
2. КОГДА администратор создает нового пользователя, ТО система ДОЛЖНА позволить назначить роль и связать с сотрудником или клиентом
3. КОГДА администратор изменяет пароль пользователя, ТО система ДОЛЖНА сохранить его в хешированном виде
4. КОГДА администратор деактивирует пользователя, ТО система ДОЛЖНА запретить ему вход в систему
5. КОГДА администратор просматривает журнал аудита, ТО система ДОЛЖНА отобразить все операции с указанием пользователя, даты и деталей изменений

### Требование 8: Интерфейс пользователя

**Пользовательская история:** Как пользователь, я хочу работать с интуитивно понятным интерфейсом, чтобы эффективно выполнять свои задачи.

#### Критерии приемки

1. КОГДА пользователь работает с приложением, ТО интерфейс ДОЛЖЕН быть адаптирован для разрешения не менее 1280x720 пикселей
2. КОГДА система отображает статусы, ТО они ДОЛЖНЫ быть выделены соответствующими цветами для быстрого восприятия
3. КОГДА пользователь выполняет операции, ТО система ДОЛЖНА предоставлять обратную связь о результате действий
4. ЕСЛИ происходит ошибка, ТО система ДОЛЖНА отобразить понятное сообщение с описанием проблемы
5. КОГДА пользователь работает с формами, ТО система ДОЛЖНА выполнять валидацию данных перед сохранением
