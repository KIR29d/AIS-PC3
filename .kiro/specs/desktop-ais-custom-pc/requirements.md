# Требования к десктопному приложению АИС "Кастом ПК"

## Введение

Автоматизированная информационная система управления продажами и сборкой кастомных ПК (АИС «Кастом ПК») представляет собой десктопное приложение на C# (.NET 8) с подключением к базе данных MySQL. Система предназначена для автоматизации процессов оформления заказов на сборку ПК, управления складскими остатками компонентов, распределения задач между сборщиками и контроля выполнения заказов.

## Требования

### Требование 1

**Пользовательская история:** Как администратор системы, я хочу иметь возможность управлять пользователями, ролями и справочниками, чтобы обеспечить корректную работу всей системы.

#### Критерии приемки

1. КОГДА администратор входит в систему ТОГДА система ДОЛЖНА предоставить доступ к модулю администрирования
2. КОГДА администратор создает нового пользователя ТОГДА система ДОЛЖНА сохранить пароль в хешированном виде (BCrypt)
3. КОГДА администратор назначает роль пользователю ТОГДА система ДОЛЖНА применить соответствующие права доступа
4. КОГДА администратор управляет справочниками ТОГДА система ДОЛЖНА обновить данные в базе данных
5. КОГДА происходят изменения данных ТОГДА система ДОЛЖНА записать информацию в журнал аудита

### Требование 2

**Пользовательская история:** Как менеджер, я хочу управлять заказами клиентов и создавать сборочные задания, чтобы обеспечить своевременное выполнение заказов.

#### Критерии приемки

1. КОГДА менеджер просматривает список заказов ТОГДА система ДОЛЖНА отобразить все заказы с их текущими статусами
2. КОГДА менеджер изменяет статус заказа ТОГДА система ДОЛЖНА обновить статус в базе данных
3. КОГДА менеджер создает сборочное задание ТОГДА система ДОЛЖНА проверить наличие компонентов на складе
4. КОГДА менеджер назначает сборщика на задание ТОГДА система ДОЛЖНА обновить информацию о назначении
5. ЕСЛИ компонентов недостаточно на складе ТОГДА система ДОЛЖНА предупредить менеджера

### Требование 3

**Пользовательская история:** Как сборщик, я хочу видеть назначенные мне задания и отмечать их выполнение, чтобы эффективно выполнять свою работу.

#### Критерии приемки

1. КОГДА сборщик входит в систему ТОГДА система ДОЛЖНА показать список его заданий
2. КОГДА сборщик начинает работу над заданием ТОГДА система ДОЛЖНА зафиксировать время начала
3. КОГДА сборщик завершает задание ТОГДА система ДОЛЖНА зафиксировать время завершения
4. КОГДА сборщик просматривает задание ТОГДА система ДОЛЖНА показать список необходимых компонентов
5. КОГДА статус задания изменяется ТОГДА система ДОЛЖНА обновить связанный заказ

### Требование 4

**Пользовательская история:** Как клиент, я хочу просматривать каталог компонентов и оформлять заказы на сборку ПК, чтобы получить желаемую конфигурацию.

#### Критерии приемки

1. КОГДА клиент просматривает каталог ТОГДА система ДОЛЖНА отобразить доступные компоненты по категориям
2. КОГДА клиент выбирает компоненты ТОГДА система ДОЛЖНА проверить их совместимость
3. КОГДА клиент оформляет заказ ТОГДА система ДОЛЖНА рассчитать общую стоимость
4. КОГДА заказ создается ТОГДА система ДОЛЖНА присвоить ему статус "новый"
5. ЕСЛИ клиент не зарегистрирован ТОГДА система ДОЛЖНА предложить регистрацию

### Требование 5

**Пользовательская история:** Как пользователь системы, я хочу иметь интуитивно понятный интерфейс в серых тонах, чтобы комфортно работать с приложением.

#### Критерии приемки

1. КОГДА пользователь запускает приложение ТОГДА система ДОЛЖНА отобразить интерфейс в серой цветовой гамме
2. КОГДА пользователь взаимодействует с элементами ТОГДА система ДОЛЖНА обеспечить четкую визуальную обратную связь
3. КОГДА загружаются изображения ТОГДА система ДОЛЖНА показать заглушки на их месте
4. КОГДА пользователь работает на сенсорном экране ТОГДА система ДОЛЖНА корректно обрабатывать сенсорный ввод
5. ЕСЛИ разрешение экрана меньше 1280x720 ТОГДА система ДОЛЖНА адаптировать интерфейс

### Требование 6

**Пользовательская история:** Как система, я должна подключаться к базе данных MySQL и обеспечивать надежное хранение данных, чтобы гарантировать целостность информации.

#### Критерии приемки

1. КОГДА приложение запускается ТОГДА система ДОЛЖНА установить соединение с MySQL через NuGet пакет
2. КОГДА выполняются операции с данными ТОГДА система ДОЛЖНА использовать транзакции для обеспечения целостности
3. КОГДА происходит ошибка подключения ТОГДА система ДОЛЖНА уведомить пользователя и предложить повторить попытку
4. КОГДА выполняются запросы ТОГДА система ДОЛЖНА использовать параметризованные запросы для защиты от SQL-инъекций
5. ЕСЛИ соединение прерывается ТОГДА система ДОЛЖНА автоматически попытаться восстановить подключение

### Требование 7

**Пользовательская история:** Как система, я должна вести полный журнал аудита всех операций, чтобы обеспечить отслеживаемость изменений.

#### Критерии приемки

1. КОГДА пользователь выполняет операцию INSERT ТОГДА система ДОЛЖНА записать информацию в таблицу аудит_изменений
2. КОГДА пользователь выполняет операцию UPDATE ТОГДА система ДОЛЖНА записать информацию в таблицу аудит_изменений
3. КОГДА пользователь выполняет операцию DELETE ТОГДА система ДОЛЖНА записать информацию в таблицу аудит_изменений
4. КОГДА записывается аудит ТОГДА система ДОЛЖНА включить ID пользователя, время операции и детали изменений
5. ЕСЛИ запись аудита не удалась ТОГДА система ДОЛЖНА отменить основную операцию

### Требование 8

**Пользовательская история:** Как система, я должна обеспечивать разграничение доступа на основе ролей пользователей, чтобы защитить конфиденциальную информацию.

#### Критерии приемки

1. КОГДА пользователь входит в систему ТОГДА система ДОЛЖНА определить его роль и предоставить соответствующий интерфейс
2. КОГДА пользователь пытается получить доступ к функции ТОГДА система ДОЛЖНА проверить его права
3. ЕСЛИ у пользователя нет прав доступа ТОГДА система ДОЛЖНА отклонить запрос и показать сообщение об ошибке
4. КОГДА администратор изменяет права пользователя ТОГДА система ДОЛЖНА немедленно применить изменения
5. КОГДА пользователь неактивен ТОГДА система ДОЛЖНА заблокировать доступ к его учетной записи
