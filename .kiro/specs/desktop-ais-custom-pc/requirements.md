# Требования к десктопному приложению АИС "Кастом ПК"

## Введение

Данный документ описывает требования к разработке десктопного приложения на C# для автоматизированной информационной системы управления продажами и сборкой кастомных ПК. Приложение должно обеспечивать полный цикл работы с заказами от оформления до отгрузки, включая управление складскими остатками и процессом сборки.

## Требования

### Требование 1: Система аутентификации и авторизации

**Пользовательская история:** Как пользователь системы, я хочу безопасно войти в систему под своей учетной записью, чтобы получить доступ к функциям согласно моей роли.

#### Критерии приемки

1. КОГДА пользователь запускает приложение, ТО система ДОЛЖНА отображать форму входа с полями логин и пароль
2. КОГДА пользователь вводит корректные учетные данные, ТО система ДОЛЖНА аутентифицировать пользователя и предоставить доступ к интерфейсу согласно его роли
3. КОГДА пользователь вводит некорректные учетные данные, ТО система ДОЛЖНА отобразить сообщение об ошибке и не предоставлять доступ
4. ЕСЛИ пользователь имеет роль "Администратор", ТО система ДОЛЖНА предоставить доступ ко всем функциям
5. ЕСЛИ пользователь имеет роль "Менеджер", ТО система ДОЛЖНА предоставить доступ к управлению заказами и клиентами
6. ЕСЛИ пользователь имеет роль "Сборщик", ТО система ДОЛЖНА предоставить доступ к сборочным заданиям и складским операциям

### Требование 2: Управление клиентами

**Пользовательская история:** Как менеджер, я хочу управлять информацией о клиентах, чтобы вести актуальную базу данных клиентов для оформления заказов.

#### Критерии приемки

1. КОГДА менеджер открывает модуль клиентов, ТО система ДОЛЖНА отобразить список всех клиентов с возможностью поиска и фильтрации
2. КОГДА менеджер создает нового клиента, ТО система ДОЛЖНА предоставить форму с полями для физического или юридического лица
3. ЕСЛИ клиент физическое лицо, ТО система ДОЛЖНА запросить фамилию, имя, отчество, email и телефон
4. ЕСЛИ клиент юридическое лицо, ТО система ДОЛЖНА запросить наименование организации, email и телефон
5. КОГДА менеджер сохраняет данные клиента, ТО система ДОЛЖНА валидировать обязательные поля и сохранить в базу данных
6. КОГДА менеджер редактирует клиента, ТО система ДОЛЖНА позволить изменить все поля кроме типа клиента

### Требование 3: Управление каталогом компонентов

**Пользовательская история:** Как администратор, я хочу управлять каталогом компонентов ПК, чтобы поддерживать актуальный список доступных для сборки компонентов.

#### Критерии приемки

1. КОГДА администратор открывает каталог компонентов, ТО система ДОЛЖНА отобразить список всех компонентов с группировкой по типам
2. КОГДА администратор добавляет новый компонент, ТО система ДОЛЖНА запросить наименование, тип, производителя
3. КОГДА администратор сохраняет компонент, ТО система ДОЛЖНА валидировать уникальность наименования и сохранить в базу
4. КОГДА администратор просматривает компонент, ТО система ДОЛЖНА показать текущие остатки по всем складам
5. КОГДА администратор удаляет компонент, ТО система ДОЛЖНА проверить отсутствие связанных заказов и остатков

### Требование 4: Оформление и управление заказами

**Пользовательская история:** Как менеджер, я хочу оформлять заказы на сборку ПК и управлять их статусами, чтобы контролировать весь процесс от создания до отгрузки.

#### Критерии приемки

1. КОГДА менеджер создает новый заказ, ТО система ДОЛЖНА позволить выбрать клиента и добавить компоненты в конфигурацию
2. КОГДА менеджер добавляет компонент в заказ, ТО система ДОЛЖНА проверить наличие на складе и рассчитать стоимость
3. КОГДА менеджер сохраняет заказ, ТО система ДОЛЖНА установить статус "новый" и сохранить в базу данных
4. КОГДА менеджер изменяет статус заказа, ТО система ДОЛЖНА валидировать переход статусов (новый → в_работе → собран → отгружен → выполнен)
5. КОГДА заказ переводится в статус "в_работе", ТО система ДОЛЖНА автоматически создать сборочное задание
6. КОГДА заказ переводится в статус "отгружен", ТО система ДОЛЖНА зафиксировать дату отгрузки

### Требование 5: Управление сборочными заданиями

**Пользовательская история:** Как сборщик, я хочу просматривать назначенные мне задания и отмечать этапы сборки, чтобы эффективно выполнять работу и информировать о прогрессе.

#### Критерии приемки

1. КОГДА сборщик входит в систему, ТО система ДОЛЖНА отобразить список заданий со статусом "ожидает" и назначенных ему заданий
2. КОГДА сборщик начинает сборку, ТО система ДОЛЖНА изменить статус задания на "в_сборке" и зафиксировать время начала
3. КОГДА сборщик просматривает задание, ТО система ДОЛЖНА показать полный список необходимых компонентов с количеством
4. КОГДА сборщик завершает сборку, ТО система ДОЛЖНА изменить статус на "готово", зафиксировать время завершения и обновить статус заказа
5. КОГДА сборщик отмечает использование компонентов, ТО система ДОЛЖНА автоматически списать их со склада

### Требование 6: Управление складскими остатками

**Пользовательская история:** Как сотрудник склада, я хочу отслеживать и обновлять остатки компонентов, чтобы обеспечить точный учет товаров.

#### Критерии приемки

1. КОГДА сотрудник открывает модуль склада, ТО система ДОЛЖНА отобразить остатки всех компонентов по складам
2. КОГДА сотрудник обновляет остаток, ТО система ДОЛЖНА позволить указать тип операции (поступление/списание) и количество
3. КОГДА сотрудник сохраняет изменение остатка, ТО система ДОЛЖНА обновить количество и зафиксировать дату операции
4. КОГДА остаток компонента становится меньше минимального, ТО система ДОЛЖНА выделить его цветом в интерфейсе
5. КОГДА происходит автоматическое списание при сборке, ТО система ДОЛЖНА обновить остатки без участия пользователя

### Требование 7: Система аудита и отчетности

**Пользовательская история:** Как администратор, я хочу просматривать журнал всех операций в системе, чтобы контролировать действия пользователей и анализировать работу.

#### Критерии приемки

1. КОГДА администратор открывает журнал аудита, ТО система ДОЛЖНА отобразить все операции с фильтрацией по дате, пользователю и типу операции
2. КОГДА происходит любое изменение данных, ТО система ДОЛЖНА автоматически записать в журнал информацию об операции
3. КОГДА администратор просматривает запись аудита, ТО система ДОЛЖНА показать детали операции включая старые и новые значения
4. КОГДА администратор формирует отчет, ТО система ДОЛЖНА предоставить данные по заказам, остаткам и производительности за период
5. КОГДА администратор экспортирует отчет, ТО система ДОЛЖНА сохранить данные в формате Excel или PDF

### Требование 8: Пользовательский интерфейс

**Пользовательская история:** Как пользователь системы, я хочу работать с интуитивно понятным интерфейсом в серых тонах, чтобы эффективно выполнять свои задачи.

#### Критерии приемки

1. КОГДА пользователь работает с приложением, ТО интерфейс ДОЛЖЕН использовать серую цветовую схему с контрастными элементами
2. КОГДА пользователь просматривает списки, ТО система ДОЛЖНА обеспечить возможность сортировки и поиска
3. КОГДА пользователь работает с формами, ТО система ДОЛЖНА валидировать данные в реальном времени
4. КОГДА система загружает данные, ТО ДОЛЖНЫ отображаться индикаторы загрузки
5. КОГДА происходит ошибка, ТО система ДОЛЖНА показать понятное сообщение с рекомендациями по устранению
6. КОГДА пользователь работает на сенсорном экране, ТО элементы интерфейса ДОЛЖНЫ быть достаточного размера для удобного использования
